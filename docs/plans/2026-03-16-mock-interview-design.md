# 模拟面试功能设计文档

**日期:** 2026-03-16
**版本:** v1.0
**状态:** 设计阶段

---

## 📋 需求概述

在现有“JD + 简历分析”的基础上，新增一个被动触发的 `模拟面试` 板块。该板块默认不主动生成内容，用户只有在查看完分析结果后，点击 `开始模拟`，系统才基于当前 `JD + 简历 + 分析结果` 生成三位角色的深度模拟问答内容。

本功能不是通用题库，而是强个性化的面板式辅导：

- 角色固定为：`直属上级`、`高级 VP`、`HRD`
- 输出形式为：每个角色 5-7 题的面试问答辅导面板
- 内容重点为：`为什么会问`、`考察点`、`回答框架`、`常见雷区`
- 不提供逐轮对话、不做自动打分、不生成标准答案全文

---

## 🎯 目标与原则

### 产品目标

- 帮助用户把“简历匹配分析”延伸到“面试准备”
- 让用户理解不同角色会如何审视这份简历
- 基于当前岗位与简历语境，提供定制化回答辅导

### 设计原则

- **被动触发**：不打断主分析流程，不默认增加等待时间
- **强个性化**：所有问题与辅导内容必须绑定当前 JD、简历和分析结果
- **角色分工清晰**：三类面试官关注点必须拉开，避免内容重复
- **只读面板优先**：第一版先提供高质量辅导面板，不扩展成对话式训练器
- **失败隔离**：模拟面试生成失败不能影响主分析和其他已有功能

---

## 👥 角色分工

### 直属上级

重点关注执行能力、项目拆解、协作落地、优先级判断和项目细节真实性。

典型问题方向：

- 你在项目中的实际职责边界是什么
- 你如何做需求判断和取舍
- 数据、用户反馈、研发约束之间如何平衡
- 某个项目结果是否真由你主导推动

### 高级 VP

重点关注业务判断、战略理解、目标拆解、跨团队影响力和高层沟通能力。

典型问题方向：

- 你如何理解岗位所在业务的增长逻辑
- 你的经历是否能支撑更高层的业务判断
- 如何从局部项目上升到组织价值
- 如何在不确定性中做关键判断

### HRD

重点关注动机、稳定性、文化适配、团队协作风险和候选人管理成熟度。

典型问题方向：

- 为什么想来这个岗位和团队
- 你的职业路径是否稳定
- 你与团队发生冲突时怎么处理
- 你对管理方式、组织节奏和变动的适应性如何

## 🧭 用户流程

### 主流程

```text
用户输入 JD 和简历
    ↓
点击“开始分析”
    ↓
主分析结果生成完成
    ↓
结果区新增“模拟面试”Tab
    ↓
用户点击“开始模拟”
    ↓
调用 /api/mock-interview
    ↓
生成三位角色的模拟面试辅导面板
```

### 交互说明

- `模拟面试` 为新的一级 Tab
- 该 Tab 默认展示说明文案和一个主按钮 `开始模拟`
- 点击按钮后进入独立 loading 状态
- 成功后在 Tab 内展示三位角色内容
- 失败后在当前 Tab 内展示错误提示和 `重新生成` 按钮

## 🧩 信息架构

### 一级 Tab

在现有 Tab 基础上新增：

- 概览
- 详细分析
- 优化建议
- HR 破冰
- `模拟面试`
- 原文

### 模拟面试 Tab 内部结构

推荐使用二级角色 Tab，而不是长列表卡片。

原因：

- 单个角色内容较长，切换比滚动更高效
- 三位角色语义清晰，天然适合分栏
- 后续若加入“重新生成单角色”也更容易扩展

结构如下：

```text
模拟面试
├── 默认态：功能说明 + 开始模拟按钮
├── 加载态：生成中提示
├── 错误态：错误提示 + 重新生成按钮
└── 成功态
    ├── 直属上级
    ├── 高级 VP
    └── HRD
```

## 📝 输出格式设计

接口返回内容建议继续使用单份 Markdown，而不是结构化 JSON。

原因：

- 现有系统已经围绕 Markdown 渲染、复制、导出构建
- 前端改动更少，复用成本最低
- 后续可先整体渲染，再逐步增强角色级导航

### 建议的 Markdown 结构

```markdown
# 模拟面试

## 直属上级

### 问题 1：......
**为什么会问**
......

**考察点**
......

**回答框架**
......

**常见雷区**
......

## 高级 VP
...

## HRD
...
```

### 单题输出规范

每道题必须固定包含以下四个模块：

- `为什么会问`
- `考察点`
- `回答框架`
- `常见雷区`

其中：

- `回答框架` 只提供作答骨架，不写成整段标准答案
- `常见雷区` 要尽量结合当前简历的真实短板，而不是泛泛提醒
- 每个角色输出 5-7 题
- 问题必须明确体现角色视角差异

### 触发逻辑

- 用户点击 `开始模拟` 后才发起请求
- 调用参数包含：
  - `jd`
  - `resume`
  - `analysis`
- 若主分析结果为空，则按钮不可用或点击后提示先完成分析

### 失效规则

当以下任一事件发生时，旧的模拟面试内容必须清空：

- 用户修改 JD
- 用户修改简历
- 用户重新点击“开始分析”

这样可以避免旧内容与新分析上下文不一致。

### UI 状态

#### 默认态

- 展示一句功能说明
- 强调会基于当前分析结果生成三个角色视角内容
- 展示 `开始模拟` 主按钮

#### 加载态

- 文案建议突出三角色生成过程
- 示例：`正在生成直属上级 / 高级 VP / HRD 视角问题...`

#### 成功态

- 渲染角色二级 Tab
- 每个角色内容支持 Markdown 展示
- 支持与现有结果区一致的复制体验

#### 错误态

- 错误提示文案
- `重新生成` 按钮
- 错误不影响其他 Tab 浏览

## 🧠 Prompt 设计

在 [lib/prompt.ts](/Users/ericcao/CascadeProjects/resume-optimizer/lib/prompt.ts) 中新增 `mockInterviewPrompt(jd, resume, analysis)`。

### Prompt 核心要求

- 明确角色为：直属上级、高级 VP、HRD
- 明确每个角色输出 5-7 题
- 明确每题固定四个模块
- 明确问题必须引用当前 JD、简历经历与分析结果中的亮点/缺口
- 明确禁止输出通用模板化面试题
- 明确不要写成长篇标准答案

### Prompt 目标

让模型输出的内容具备以下特征：

- 有真实岗位针对性
- 有具体面试官视角
- 有实战作答指导价值
- 三个角色之间低重复

## 🔒 错误处理与隔离

### 独立失败原则

模拟面试是二阶段增强能力，不是主链路能力。

因此：

- 主分析成功后，即使模拟面试失败，现有分析结果也必须完整可用
- 模拟面试接口失败时，只影响 `模拟面试` Tab
- 不允许因为模拟面试失败而回退、清空或覆盖主分析结果

### 推荐错误文案

- `模拟面试生成失败，请稍后重试。`
- `当前分析结果不可用，请先重新生成简历分析。`

## ✅ 测试要点

### 状态流转

- 首次进入 `模拟面试` 时为默认态
- 点击 `开始模拟` 后进入加载态
- 成功后正确显示三角色内容
- 失败后正确显示错误态
- 点击 `重新生成` 后可再次请求

### 数据一致性

- 重新分析后旧的模拟内容被清空
- 修改 JD / 简历后旧的模拟内容被清空
- 无主分析结果时不能生成模拟面试

### 输出结构稳定性

- Markdown 中必须包含：
  - `# 模拟面试`
  - `## 直属上级`
  - `## 高级 VP`
  - `## HRD`
- 每个角色必须包含 5-7 题
- 每题必须带四个固定模块

### 体验验证

- 移动端长内容可读性正常
- 角色切换体验顺滑
- 复制逻辑不会复制空内容
- 导出原逻辑不被破坏

## 🚫 第一版不做的内容

- 不做逐轮聊天式面试
- 不做用户答题后 AI 追问
- 不做打分、点评、胜任力画像
- 不做录音、语音转文字
- 不做历史记录和多轮会话管理
- 不做单角色增量生成

这些能力都可能是后续迭代方向，但不应进入第一版。
