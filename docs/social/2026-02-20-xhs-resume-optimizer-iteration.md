## 3) 5页图文脚本（小红书卡片）

## 第1页｜开场（问题）
标题：这个项目，之前“看起来能用”，其实不太能用
正文：
- 我做了一个 AI 简历优化器，最初版本可以分析 JD 和简历
- 但真实使用时有两个关键问题：
  - Tab 点了几乎没法快速定位内容
  - “HR 破冰文案”这个高价值功能没有跑通
- 这轮迭代目标很明确：把“功能展示”升级成“投递闭环”

## 第2页｜核心修复1（信息结构）
标题：Tab 从“装饰”变成“导航”
正文：
- 之前：切换 Tab 但内容不变，用户要自己翻全文
- 现在：按 Markdown 标题做映射和回退
  - 精确匹配：概览 / 详细分析 / 优化建议
  - 关键词回退：避免模型输出波动导致空白
  - 最终兜底：无匹配时返回全文，保证不空屏
- 结果：信息查找速度提升，阅读路径更清晰

## 第3页｜核心修复2（转化动作）
标题：补上“HR 破冰文案”能力
正文：
- 新增独立接口：`/api/icebreaker`
- 串行流程：分析完成后自动生成 100 字内破冰话术
- 加了加载态、失败提示、不中断主流程
- 实际踩坑：模型参数限制导致接口报错
- 处理：修正模型参数后稳定返回
- 价值：从“知道怎么改简历”到“能直接开聊 HR”

## 第4页｜体验优化（细节决定可用性）
标题：我顺手补了这几个高频细节
正文：
- 每个 Tab 支持一键复制当前内容
- 粘贴失败 / 剪贴板为空时有明确提示
- 去掉不安全的 Markdown 原始 HTML 渲染路径
- 生产构建去除外网字体依赖，避免构建失败
- 文档和实现对齐（服务端 API Key 方案）

## 第5页｜结果与复盘（方法论）
标题：这轮迭代给我的 3 个结论
正文：
- 1）先修“信息定位”，再谈“内容质量”
- 2）高价值功能一定要闭环：生成 -> 可见 -> 可复制 -> 可执行
- 3）AI 产品不是“调 prompt”，而是“端到端可用性工程”
- 下一步：做破冰文案多版本 + A/B 测试 + 投递反馈回流

## 4) 发布正文（可直接发）
这周把我的「AI 简历优化器」做了一轮很务实的迭代。

一开始它“能分析”，但不够“能用”：
1）Tab 切换不能快速定位；
2）HR 破冰文案功能缺失或失败。

这次重点做了三件事：
- 按 Markdown 标题做内容分栏（并加关键词回退，避免空白）
- 新增 HR 破冰文案独立接口，分析后自动生成可用话术
- 补齐复制、粘贴反馈、安全渲染、构建稳定性等细节

我越来越确定：
AI 产品的关键不只是模型效果，而是从输入到动作的可执行闭环。
